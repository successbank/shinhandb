## 2. 이전 시도 및 실패 원인

### 1차 수정 (실패)
- **가설:** DOM reordering 및 비결정적 정렬
- **조치:** DOM reordering 삭제, deterministic sort 적용
- **결과:** 효과 없음

### 2차 수정 (실패)
- **가설:** CSS/JS 로딩 타이밍 차이
- **조치:** CSS/JS를 조건부 렌더링 밖으로 이동, `swiperCssLoaded` 가드, DOM readiness 체크, `rewind: false`
- **결과:** 효과 없음

### 왜 실패했는가
1차, 2차 모두 **"슬라이드 DOM 순서"** 또는 **"Swiper 라이브러리 로딩 타이밍"**을 원인으로 가정했으나, 실제 문제는 **슬라이드 내부 이미지의 로딩 타이밍**이었음. DOM 순서는 항상 정확했고, Swiper JS/CSS도 정상 로딩되었으나, **이미지가 아직 로딩되지 않은 슬라이드의 크기가 0px**이 되는 것이 진짜 원인이었음.

## 3. 근본 원인 분석 (3차)

### 핵심 메커니즘

```
Swiper Coverflow 3D transform 계산식:
slideProgress = (translate + slideSize/2 - center) / slideSize

→ slideSize = 0px (이미지 미로딩) 일 때:
  slideProgress = NaN 또는 Infinity
  → 슬라이드가 겹치거나 잘못된 위치에 배치
  → 사용자에게 "순서가 뒤바뀐 것"으로 인식
```

### 발생 조건

| 조건 | 설명 |
|------|------|
| **Swiper 설정** | `slidesPerView: 'auto'` + CSS `width: auto !important` |
| **이미지 미로딩** | `<img>` 태그의 `naturalWidth = 0` → 슬라이드 `offsetWidth = 0px` |
| **Coverflow 계산** | `slideSize = 0`이면 3D transform에 NaN/Infinity 발생 |

### 브라우저별 차이 원인

각 브라우저(Chrome, Firefox, Safari)의 **이미지 캐시 전략과 로딩 우선순위가 다름**:
- Chrome: 뷰포트 근처 이미지 우선 로딩 (Lazy loading 최적화)
- Firefox: 순차 로딩 경향
- Safari: 독자적 캐시 정책

→ **어떤 슬라이드가 0px인지가 브라우저마다 달라짐** → 순서가 다르게 보임

### 재클릭 시 정상 동작 이유

첫 클릭에서 모든 이미지가 브라우저 캐시에 저장됨 → 재클릭 시 이미지가 즉시 로딩 → `offsetWidth` 즉시 확정 → Swiper 계산 정상

## 4. 수정 내용

### Step 1: 이미지 로딩 완료 후 Swiper 초기화 (핵심)

**위치:** Swiper 초기화 `useEffect` (line 390~526)

기존 double-rAF 방식은 DOM 렌더링만 대기하고 이미지 로딩은 대기하지 않았음. **모든 슬라이드 이미지의 `load`/`error` 이벤트를 대기한 후 Swiper를 생성**하는 패턴으로 전면 교체.

```typescript
// 핵심 로직: 이미지 로딩 대기
const slideImages = container.querySelectorAll('.swiper-slide img');
const imageLoadPromises = Array.from(slideImages).map(img => {
  const htmlImg = img as HTMLImageElement;
  if (htmlImg.complete && htmlImg.naturalWidth > 0) return Promise.resolve();
  return new Promise<void>(resolve => {
    const onDone = () => { /* cleanup + resolve */ };
    htmlImg.addEventListener('load', onDone);
    htmlImg.addEventListener('error', onDone);
  });
});

// 5초 타임아웃 안전장치
await Promise.race([
  Promise.all(imageLoadPromises),
  new Promise<void>(r => setTimeout(r, 5000))
]);

// 이미지 모두 로딩 후 Swiper 생성
createSwiperInstance();
```

**안전장치:**
- `cancelled` 플래그: 컴포넌트 언마운트 시 비동기 작업 중단
- 5초 타임아웃: 네트워크 장애 시에도 Swiper 생성 보장
- `error` 이벤트 핸들링: 이미지 로딩 실패 시에도 진행

---

### Step 2: CSS min-width/min-height 추가 (0px 방지 fallback)

**위치:** `<style jsx>` 블록

이미지 로딩 전에도 슬라이드가 최소 크기를 갖도록 보장하는 **방어적 CSS**.

| Breakpoint | min-width | min-height |
|------------|-----------|------------|
| 기본 (Desktop) | 200px | 280px |
| ≤768px (Tablet) | 180px | 250px |
| ≤480px (Mobile) | 160px | 220px |

이미지 로딩 피드백을 위한 미세한 배경색도 추가:
```css
.quarter-swiper .swiper-slide img {
  background: rgba(255,255,255,0.05);
}
```

---

### Step 3: `normalizeSlideIndex: false` 추가

**위치:** Swiper 옵션

```typescript
swiperRef.current = new Swiper('.quarter-swiper', {
  // ...
  normalizeSlideIndex: false,  // 인덱스 정규화 비활성화
  // ...
});
```

**근거:** Swiper GitHub Issue #5075 - `centeredSlides` + `slidesPerView: 'auto'` 조합에서 `normalizeSlideIndex`가 잘못된 초기 슬라이드 인덱스를 계산하는 알려진 문제.

---

### Step 4: 프론트엔드 정렬 기준을 백엔드와 동일하게

**위치:** 신한금융지주/신한은행 분기 클릭 핸들러 (line 913, 1017)

```typescript
// 변경 전: displayOrder → projectId (2단계)
// 변경 후: displayOrder → createdAt → projectId (3단계, 백엔드와 동일)
projects: [...projects].sort((a, b) => {
  const orderDiff = (a.displayOrder ?? 999) - (b.displayOrder ?? 999);
  if (orderDiff !== 0) return orderDiff;
  const dateDiff = new Date(a.createdAt).getTime() - new Date(b.createdAt).getTime();
  if (dateDiff !== 0) return dateDiff;
  return (a.projectId || '').localeCompare(b.projectId || '');
})
```

**이유:** `displayOrder`가 동일한 프로젝트 간 정렬이 비결정적이었음. `createdAt` 기준 추가로 백엔드 정렬과 100% 일치.

## 8. 교훈 (Lessons Learned)

### 기술적 교훈

1. **Swiper `slidesPerView: 'auto'`는 슬라이드 크기에 의존한다.**
   - 이미지 기반 슬라이드에서는 반드시 이미지 로딩 완료 후 초기화해야 함.
   - CSS `min-width`로 방어적 최소 크기를 설정하는 것이 안전.

2. **브라우저별 이미지 캐시/로딩 전략이 다르다.**
   - "같은 코드인데 브라우저마다 다르다" → 네트워크/렌더링 타이밍을 의심할 것.
   - 특히 3D transform 기반 라이브러리는 크기 계산에 민감.

3. **double-rAF는 DOM 렌더링만 보장하고, 이미지 로딩은 보장하지 않는다.**
   - `requestAnimationFrame` × 2 ≠ 이미지 로딩 완료
   - `HTMLImageElement.complete` + `load` 이벤트 조합 필요.

### 프로세스 교훈

1. **1, 2차 실패 후 가설을 재검토한 것이 핵심.**
   - 같은 방향의 추가 수정이 아닌, 근본 원인 재분석이 필요했음.
   - 리서치팀(박지원)의 Swiper 소스코드 분석이 결정적 단서 제공.

2. **"재클릭 시 정상"이라는 단서가 핵심 힌트.**
   - 캐시 유무에 따른 동작 차이 → 이미지 로딩 타이밍 문제 확정.
